# 嘟嘟桌宠 — 优先级评估与阶段规划

## 优先级矩阵

用两个维度评估：**风险**（不确定性高 → 先做）+ **依赖**（被依赖 → 先做）。

| 优先级 | 任务 | 风险 | 依赖关系 | 为什么排这里 |
|--------|------|------|---------|-------------|
| P0 | Godot 桌面透明悬浮窗口 | **高** | 整个 Godot 方案的前置条件 | 搞不通就得换 Electron/Tauri/PyQt，先踩坑 |
| P0 | Python ↔ Godot WebSocket 通信 | 中 | 所有功能的前置条件 | 后端和前端之间的唯一桥梁，协议定错后面全改 |
| P1 | 消息协议定义 | 低 | WebSocket 通了才能验证 | 协议是前后端的 contract，早定早安心 |
| P1 | 精灵渲染 + 动画状态机 | 中 | 需要 P0 窗口 + P0 通信 | 猫能动起来才叫宠物 |
| P1 | AI 对话核心（上下文 + API 调用） | 低 | 依赖 WebSocket | 从 Nero 直接移植，技术成熟 |
| P2 | 上下文压缩 | 低 | 依赖对话核心 | 从 Nero 移植算法，无新风险 |
| P2 | 气泡对话 UI | 低 | 依赖精灵渲染 | Godot 做这个很成熟 |
| P2 | Poisson 心跳（主动卖萌） | 低 | 依赖对话核心 | 从 Nero 移植，纯算法 |
| P2 | 番茄钟 | 低 | 依赖 WebSocket + 气泡 | 计时在 Python 侧，Godot 只负责动画 |
| P3 | 喝水/提肛提醒 | 低 | 同番茄钟 | 和番茄钟同模式，复用 |
| P3 | 历史对话窗口 | 低 | 依赖对话核心 | 独立 UI 模块 |
| P4 | 星盘 | 中 | 无依赖 | 全新功能，放到 v2 |
| P4 | 换装系统 | 中 | 依赖精灵系统 | 需要考虑精灵切换架构 |
| P4 | 喂食互动 | 中 | 依赖动画系统 | 新交互模式 |
| P4 | 小屋系统 | 高 | 依赖精灵 + 通信 | Nero house 的简化移植 |

## 与另一个 AI 评估的差异

另一个 AI 的评估整体合理，但有几点不同意见：

1. **消息协议应该独立作为 P1 里程碑**——它不是 WebSocket "通了"之后顺带做的事。协议是前后端的 contract，包括：消息类型枚举、错误处理约定、心跳/ping 机制、重连策略。协议定了之后，前后端可以并行开发。

2. **P0 的两个验证是短路条件**——如果 Godot 透明窗口搞不定，整个方案降级。应该在 1-2 天内出结论，不要拖。

3. **动画系统的优先级应该比对话更靠前**——因为嘟嘟首先是"宠物"然后才是"AI"。猫能动（idle 呼吸、走路颠、肚子摊地上），即使对话还没接上，产品已经是可感知的。对话是给猫赋予灵魂，但先得有肉体。

4. **错误恢复应该在每个阶段都考虑**——Python 后端挂了，Godot 不能白屏。这是容易被忽略但用户体验极差的情况。应该在协议里就定义 `backend_status` 消息。
